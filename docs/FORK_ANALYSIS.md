## Per-branch detail (diverged branches with novel content)

### lsl9119 : fix/search-by-amount-crash-and-batch-timeout  (analysed 2026-06-07, head `67e8f51`)

- What diverged: a single commit, "fix: prevent server crashes in search_by_amount and update_batch".
- Novel functionality upstream lacks:
  - **Unbounded-scan guard in `actual_transactions_search_by_amount`**
    (`src/tools/transactions_search_by_amount.ts`): rejects a search that has no
    `accountId` and no date range, because an unfiltered amount search scans the
    whole transaction database and can exhaust memory / crash the server. Also
    wraps the handler in try/catch and improves the account-name-vs-UUID error.
    Idea value: HIGH. This is a reliability/DoS-hardening fix in a tool we ship,
    and upstream's `transactions_search_by_amount` currently has no such guard.
    Note: during a recent `/local-env` run upstream emitted a non-actionable
    "Cannot destructure property 'transfer_acct' of null" on a search path, so
    there may be a related robustness gap worth checking at the same time.
  - **Batch-timeout hardening in `actual_transactions_update_batch`**
    (`src/tools/transactions_update_batch.ts`): guards against the batch update
    crashing/timing out. Idea value: medium-high (reliability). Confirm whether
    our kill-switch / batch handling already covers this before adopting.
- Noise (ignored): none; it is a focused two-file fix.

### ahmadrazach : main  (analysed 2026-06-07, head `8460876`)

- Novel functionality upstream lacks:

### ZanzyTHEbar : main  (analysed 2026-06-07, head `f555255`)

The largest fork (315 commits ahead, 228 files). A large share of those commits
are NOT features (type-hardening "replace 'as any'", schema/test refinement, CI,
merges of the owner's own roadmap branches); those are noise. The genuinely novel
subsystems are:

- **Semantic / hybrid search over budget data** (`src/lib/search/*`, tools
  `hybrid_search.ts`, `search_similar.ts`, `search_index_info.ts`): embeddings-based
  search with a hybrid keyword+vector engine, an embedding pipeline + index + sync
  state, query expansion/analysis, and a response cache with invalidation.
  Pluggable providers: Ollama (local), HuggingFace (local), OpenAI-compatible.
  Idea value: HIGH. Effort: LARGE. No upstream equivalent (closest is
  `actual_query_run`, structured-only).
- **Pluggable auth with an LDAP provider** (`src/auth/auth-factory.ts`,
  `ldap-provider.ts`, `oidc-provider.ts`, `auth-middleware.ts`, `types.ts`): an
  auth-factory over multiple providers, adding LDAP alongside OIDC. Idea value:
  medium. Upstream has OIDC + static bearer, no LDAP and no provider factory.
- **Worker-thread session model + write coordinator**
  (`src/lib/SessionWorkerManager.ts`, `WriteCoordinator.ts`,
  `src/workers/actualSessionWorker.ts`): Actual sessions in worker threads with a
  write coordinator. Idea value: medium-high and architecturally relevant as a
  different answer to the session/concurrency problem upstream solves with
  `ActualConnectionPool` + `withApiLock` (see #134, #173). Effort: LARGE, high
  risk. Worth studying as a design reference.
- **OpenAPI-driven tool generation** (`scripts/generate-tools.ts`,
  `scripts/openapi/actual-openapi.yaml`, a `generator-check` CI): generates tool
  stubs from an OpenAPI spec and verifies they stay in sync. Idea value: medium.
  Upstream generates client types but not tool stubs.
- **Meta-tool registry + dynamic dispatch** (`src/tools/meta_registry.ts`,
  `meta_tool_call.ts`): one meta-tool lists and invokes others dynamically. Idea
  value: medium (helps clients that cap tool counts).
- Lower value / likely superseded: `toolNameNormalization.ts` (dots to underscores
  for OpenAI compat; our names already use underscores), a standalone
  `src/server/sseServer.ts` (our StreamableHTTP already carries SSE), and helper
  modules (`cachedRefs`, `lookupByName`, `budgetContext`, `wrapToolCall`,
  `toolResult`) that partly overlap our adapter / `actual_get_id_by_name` / toolFactory.

## How this file is maintained

Run the `fork-analysis` skill (ask to "analyse the forks" or "scan forks for
features"). It enumerates every fork and each meaningful branch (excluding
`dependabot/*`), re-checks divergence, re-analyses only branches whose head sha
changed since the rows above, carries forward unchanged verdicts, and rewrites
this file. It never clones or runs fork code; it reads the GitHub API and file
contents only.
