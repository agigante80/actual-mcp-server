## Per-branch detail (diverged branches with novel content)

### dsqr84 : develop  (analysed 2026-06-17, head `95ee4e3`)

- What diverged: five focused OIDC fixes by @dsqr84.
- Novel functionality upstream lacks (ticketed as #244):
  - **JWKS discovery via the OIDC discovery document** (`src/server/httpServer.ts`): fetches
    `jwks_uri` from `${OIDC_ISSUER}/.well-known/openid-configuration` instead of our hardcoded
    `${OIDC_ISSUER}/.well-known/jwks`, so Authentik, Keycloak, Auth0, and Casdoor all work
    regardless of their JWKS path. Plus more flexible audience validation (resource URI or raw
    client id). Idea value: HIGH (real interop sharp edge).
  - **Expired/invalid JWT no longer crashes the server** (`httpServer.ts`,
    `src/lib/rejection-allowlist.ts`): wraps the error so `mcp-auth` returns a clean 401, with a
    rejection-allowlist entry so it cannot become an unhandled rejection. Adds
    `tests/unit/httpServer_oidc_token_expiry.test.js`. Idea value: HIGH (crash/robustness fix in
    the auth path).
  - **Per-session JWT validation caching** and **sessionPrincipals cleanup on failed init**.
    Idea value: medium; the caching needs a careful security review so an expired/revoked token
    cannot keep a session alive (must not weaken the #163 per-request principal check).
- Noise (ignored): `docker-compose.yaml` and `package.json` tweaks.
- `dsqr84:main` (head `f7cfcb3`, 3 ahead) is an older subset of these commits; covered by #244.

### plainprogrammer : claude/beautiful-babbage-8kkzrh  (analysed 2026-06-17, head `2361cf2`)

- What diverged: a single doc commit adding `docs/SECURITY_ASSESSMENT.md`, a personal security
  review of v0.6.50. Not functionality, so not ticketed as a feature.
- Worth noting (not a feature ticket): its top finding (H-1, HTTP unauthenticated by default and
  binds all interfaces) is exactly what #242 fixed in v0.7.9, so it is already addressed. Its
  other observations (irreversible deletes with no confirmation gate, unencrypted local cache,
  prompt-injection blast radius) are known/inherent. The "destructive-op confirmation gate" idea
  for the delete tools could become a separate feature discussion if desired, but it is an
  opinion in a doc, not implemented fork code, so it is recorded here rather than auto-ticketed.

### palhatlem : patch-1  (analysed 2026-06-17, head `d5b365b`)

- What diverged: `fix(tools): give transactions_import txs an explicit array schema`
  (`src/tools/transactions_import.ts`).
- Verdict: SUPERSEDED. Upstream already shipped the same change as part of #217 (our
  `transactions_import` already defines an explicit `z.array(...)` item schema with date/amount/
  payee/etc., replacing the old `z.unknown()` that serialised to an empty `{}` for stdio clients).
  The fork patch is effectively identical in intent. No ticket.

## How this file is maintained

Run the `fork-analysis` skill (ask to "analyse the forks" or "scan forks for
features"). It enumerates every fork and each meaningful branch (excluding
`dependabot/*`), re-checks divergence, re-analyses only branches whose head sha
changed since the rows above, carries forward unchanged verdicts, and rewrites
this file. It never clones or runs fork code; it reads the GitHub API and file
contents only.
